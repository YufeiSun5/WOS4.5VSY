# 1.1. 开发仓库

用户在业务层面用仓库管理开发数据，仓库下有子仓库、模型、拷贝、别名、资源对象。
模型库提供ID固定为0的系统仓库，存储在默认数据区0中。系统仓库本身不可查询、不可修改、不可删除。但可以作为范围查询其包含的内容。
创建子仓库时需要指定父仓库，创建根仓库时指定的父仓库是系统仓库0。同层级的仓库数量数量不做限制。仓库层级数量不做限制。

# 1.2. 模型

一种类型的对象有相同的数据结构、业务逻辑。其数据结构、业务逻辑是该类对象的基因，也就是模型。WOS通过先构建基因（模型），再批量实例化（创建拷贝）的方法，实现快速建模能力。
WOS4.5的模型分为编辑版本和历史版本。
模型在新建时会产生1个编辑版本。编辑版本只有1份，只要模型存在，编辑版本则一定存在。编辑版本的版本号为0。
历史版本通过提交函数生成，提交功能是将编辑版本0当前的数据额外存储一份历史数据。历史版本的版本号为正数，从1开始递增且不回收。
模型只有编辑版本可以修改，历史版本在产生后不会变化（除name、displayname、label、repositoryGUID属性，这些属性会随编辑版本修改而变化）。
模型只有历史版本可以产生拷贝。
每个历史版本可以独立删除，编辑版本不能独立删除，删除编辑版本0表示删除整个模型包括所有历史版本。
查询模型时，支持输入版本号-1表示查询所有历史版本，不包括编辑版本。

# 1.3.2.1.3. 域型拷贝

DTE模型把系统视为主体的集合和客体的集合。主体的集合被称为域，客体的集合被称为型。域和型是相对的，同一个对象作为主体时属于域，作为客体使用时则属于型。
通过配置域对各个型的访问权限，描述主体对客体的访问权限。
域型的集合可以建立细分域型，细分域继承父域对各个型的访问权限。若域A对型B具有访问权限，则域A的细分域也有型B的权限，域A也有型B的细分型的权限。

仓库数据包是开发仓库的数据压缩包。

 仓库数据包直接存储在数据区中。开发仓库打包生成的仓库数据包ID和GUID与其完全一致，每次打包时会生成版本号，版本号从1开始递增且不回收。

# 2.5.1. 部署状态

模型库的部署状态有6个静态的状态：未部署、部分已部署、已部署、已启动、部分已启动、已停止。如果五个子系统的状态存在冲突，模型库按走的最远的子系统显示，已启动>已停止>已部署>未部署。如实时库未部署、历史库已部署、计划库已启动，模型库将显示部分已启动。
模型库的部署状态有4个进行中的状态：反部署中、部署中、启动中、停止中。当模型库正在进行反部署、部署、启动、停止操作时，对象的状态会变成对应的进行中状态。处于进行中状态的对象无法开启新的任务，必须等待当前任务进行完整才能进行下一步操作。
运行库的部署状态只有4个静态的状态：未部署、已部署、已启动、已停止。

# 2.5.2. 可靠性

部署启停功能需要保证可靠性，五个子系统的操作结果可能存在差异，但是不允许出现模型库统计的状态与运行库实际状态不相符的情况。例如模型库发起部署，实时库部署完成，但是汇报结果时网络出现异常，模型库未统计到实时库的部署结果，而实时库已部署。
若出现异常使用情况（如用户单独清空某个运行库），下次进行部署启停操作时，需要更新为最新的状态。
对象在任何静态状态下都可以发起部署启停四种操作，各子系统根据本库的数据进行该操作。如果目标已经处于部署状态，再进行部署，返回成功。（其他操作同理）

1、实时库未部署、历史库已部署、计划库已启动，模型库可以发起部署功能，最终模型库返回部分成功，实时库成功，历史库成功，计划库失败（对象已启动）。此时模型库和所有运行库的部署状态均为已部署（已启动和已停止的状态不变）。
2、实时库未部署、历史库已启动、计划库已停止，模型库可以发起启动功能，最终模型库返回部分成功，实时库失败（对象未部署），历史库成功，计划库成功。此时模型库部署状态为部分已启动。实时库、历史库、计划库分别显示未部署、已启动、已启动。
3、模型库时空对象下有2个已部署时空仓库，2个未部署时空仓库。启动时空对象，最终时空对象已启动，2个已部署时空仓库已启动，2个未部署时空仓库维持原状态。模型库启动返回部分成功，运行库返回部分成功。
