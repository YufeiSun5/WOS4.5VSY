# 1. 概述

# 1.1. App引擎

App引擎：即App计算引擎。
App引擎是WOS系统的计算执行服务，完成App业务功能的调度执行。主要负责App对象的创建（部署启动）、删除（反部署停止）、查询、修改、订阅；App调试、功能调用、接口函数调用、定时调用、事务控制等。
根据支持的语言类型分为：元语言App引擎、WebJS App引擎、NodeJS App引擎等。

## 1.1. App引擎

# 1.2. App对象及分类

App对象：指通过App拷贝（或App模型）创建在App引擎中的运行对象，包含业务功能执行的逻辑脚本（程序代码）。App对象创建后自动执行脚本代码，从而完成用户的业务功能。
App对象按照运行依赖的App引擎类型及语言类型，分为：元语言App、WebJS App、NodeJS App等。

# 2.2.1. 创建App对象

支持根据App拷贝创建根App、子App对象，创建即启动运行。
App拷贝可以来自本地，也可以来自远程WOS（如：云WOS中的App拷贝部署到端WOS）。
子App只能由父App创建。App对象按照父子结构进行组织，形成一棵App对象树。
![](./media/创建App对象.png)
**所属时空（部署时空）：**
所属时空也叫部署时空，表示App对象创建的本地时空。端App对象的所属时空为端时空，云App对象的所属时空为云时空。
BS模式下，WebJS App对象看做云App对象，所属时空为云时空；
CS模式下，WebJS App对象看做端App对象，所属时空为端时空。
**远程访问时空：**
App对象针对每一个远程WOS都可以配置一个远程访问时空。当App对象登录远程WOS后，就可以访问远程访问时空中的App对象和数据对象。
BS模式下，WebJS App可以访问所属云时空的数据；
CS模式下，WebJS App对象访问云上数据，需要配置云上对应时空才能访问。
**App的用户登录：**
创建App对象时支持配置用户列表，用户登录成功后，才可以访问本地WOS数据，或访问远程WOS数据。具体详见【WOS互联】章节，在此不再赘述。

**运行状态推送说明：**
启动中：根App对象开始创建，推送启动中状态，表示开始启动。

运行正常：根App对象创建成功执行完onCreate函数后，推送运行正常状态，表示启动成功。

启动异常：如果执行onCreate()出现异常，推送启动异常状态，不需要执行onDestroy；（用户需要从桌面删除、修正代码重新创建）。

停止中：根App对象开始删除，推送停止中状态。

已停止：删除成功执行完onDestroy()函数后，推送根App对象已停止状态。

停止异常：如果执行onDestroy()异常，推送停止异常状态；（用户需要从桌面重新删除，执行强制删除，不会执行onDestroy()）。

运行异常：如果根App对象运行过程中出现异常情况（堆栈溢出等），推送运行异常状态。运行异常不影响App对象的脚本继续执行。
**App运行过程产生报警的事件：**
App启动失败

App停止失败

App运行异常（OnCreate执行完成后，运行过程脚本异常或其他异常）

单次任务耗时超过阈值设置（阈值默认基于平均耗时m设置，[m+2\*标准差，m-2\*标准差]，下限一般忽略不会产生报警）。

# 2.2.7. 接口调用

App对象之间可以进行接口调用，即调用App中定义的自定义功能函数。
接口调用主要应用于前端调后台的场景中，即客户端上App对象调用后台（云）上App对象的接口函数，后台（云）App对象接口函数进行后台数据计算处理，再将计算处理后的结果返回给调用的App对象。
后台（云）App对象之间也支持接口调用。
**不支持的场景：**
不支持WebJS App对象之间接口调用。
**接口调用的用户权限：**
App之间接口调用时，接口函数中执行脚本使用调用方App的用户权限（包括业务、安全、审计、运维）。
前端App调用后台App1（如WebJSApp调用后台元语言App），使用前端App用户权限；如果后台App1继续调用后台App2的情况，则使用后台App1的用户权限。

# 2.2.10. 时空约束

为了业务程序之间彼此隔离，App引擎中的对象操作增加时空约束，限制写操作（创建、删除、接口调用App）不能跨时空，读操作（查询、订阅）可以跨时空。
**后台App的时空约束：**
后台App，主要指元语言App、NodeJS App等，操作App对象的时空约束规则如下：
创建、删除：不允许跨时空，只能操作本时空内的App；
修改：后台App不支持修改操作；
接口调用：不允许跨时空，本地调用时，只能调用同一时空内的App，远程调用时，只能调用其远程访问时空列表内的App；
查询、订阅：没有时空约束；
**前端App的时空约束：**
前端App，主要指WebJS App，操作App对象的时空约束规则如下：
创建、删除、修改：不允许跨时空，只能操作本时空内的App；

修改：后台App不支持修改操作；

接口调用：不允许跨时空，本地调用时，只能调用同一时空内的App，远程调用时，只能调用其远程访问时空列表内的App；

BS模式下，WebJS App对象看做云App对象，调用云后台App是本地调用，按所属时空约束；

CS模式下，WebJS App对象看做端App对象，调用云后台App是远程调用，按访问时空约束。

查询、订阅：没有时空约束。

 接口调用：不允许跨时空，本地调用时，只能调用同一时空内的App，远程调用时，只能调用其远程访问时空列表内的App；

 查询、订阅：没有时空约束；

 前端App，主要指WebJS App，操作App对象的时空约束规则如下：

 BS模式下，WebJS App对象看做云App对象，调用云后台App是本地调用，按所属时空约束；

# 2.7. 事务

WOS4.0支持两种事务隔离级别：读已提交、可重复读，默认为可重复读。
**读已提交（read committed）：**读操作读的是对象最后已提交入库的数据，而不是事务里未提交的数据。可以保证读取的是已经提交过的数据。避免脏读，但会出现不可重复读等。
**可重复读（repeatable read）：**事务开启后，读过的数据，不允许被事务外操作修改，保证事务里前后读到的数据一致。避免脏读，避免不可重复读。
**App引擎提供的事务操作：**
开启事务
查询事务ID
提交事务
回滚事务
**应用场景一：**
需要向实时库插入2条数据，同时向历史库插入同样的2条数据，这2条数据是一个整体，要么全部插入成功，要么都插入不成功。
在向实时库和历史库插入2条数据前开启事务，执行插入操作，如果插入操作无异常，进行事务提交，当前事务结束。如果插入操作有异常，进行事务回滚，回滚到开启事务前的状态，当前事务结束。
**应用场景二：**
边仓库向中心仓库领了一批物料，边仓库增加了一批物料，中心仓库减少了一批物料，两边必须都操作成功或都操作失败。
在边仓库增加一批物料操作前开启事务，执行增加操作，再向中心仓库减少一批物料，执行减少操作。增减操作无异常时提交事务，当前事务结束。如果增减操作有异常，进行事务回滚，回滚到开启事务前的状态，当前事务结束。

# 3.4. 多副本及永不停机

App引擎服务和其他子系统服务一致，都采用集群多副本（多节点）架构，即：每个副本上可以运行一个App引擎服务实例，每个服务下有多个数据区。在某个副本上有一个数据区（主），则在其他副本上都有一个对应的数据区（副）。数据区主、副之间会进行数据同步，从而在故障发生时进行切换。
![](./media/多副本.png)
**多副本运行机制：**
主副本数据区内App对象启动成功，即：onCreate()执行完毕，生成了定时任务、订阅任务。此时多副本App对象之间进行数据同步，保证其他副本的App对象维持定时和订阅任务，处于未被激活的状态。
一旦主副本节点发生故障，切换到其他某个副本，此时该副本内App对象被激活，副本数据区内App对象不会再执行onCreate()函数，只是定时和订阅任务存在，可以继续处理定时回调和订阅回调事件。从而达到多副本切换后App对象继续正常运行并处理任务的效果。
**多副本切换支持容灾恢复、无扰升级：**
场景1：主节点断电或断网或服务异常崩溃后，自动切换到从节点，App对象在从节点上正常运行。之后主节点恢复，可以手动切换回来，App对象又接着在主节点上正常运行。
场景2：主节点要升级wos，手动切换到从节点，App对象在从节点上正常运行。主节点升级wos成功后，手动切换回来，App对象又接着在主节点上正常运行。然后从节点再升级wos。
**永不停机：**
永不停机指在WOS系统升级过程中，对外服务不中断，用户无感知，从而实现高可用和在线更新。
**升级流程说明：**
假设当前WOS系统运行系统版本为V1，要升级到V2，且V1和V2前后兼容（V2能读V1的数据，V1能接收V2的消息等）。
步骤1：从机升级
1）停止从机
2）备份数据（用于回滚）。
3）将从机升级到V2版本。
4）启动从机V2，开始从主机V1同步数据。
步骤2：从机切换为主机
步骤3：原主机切换为从机
1）停止主机
2）备份数据（用于回滚）。
3）升级到V2版本。
4）启动后作为V2从机，反向同步新主机的数据。
