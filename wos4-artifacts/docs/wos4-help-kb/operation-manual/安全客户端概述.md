# 2.7锁和钥匙

在WOS4.0中，主体对客体的权限需要通过三层权限校验：
通过级别和域型的校验；
通过范围准入权限的校验；
通过操作权限的校验。
其中范围准入权限和操作权限就是通过锁和钥匙的判定。系统会提供默认钥匙，除此之外，用户可以自定义锁。钥匙不需要用户感知，创建锁的时候会同时创建一把钥匙。多个对象可以加同一把锁，创建对象时默认加了读锁、写锁以及删除锁。 
以下是锁的类型：
安全锁
控制主体加锁权限、分配钥匙的权限以及对 【安全标记】的访问和操作权限
审计锁
控制主体【数据对象】的访问和操作权限
权限锁 - 读
控制主体对客体的读（或者查询）权限
权限锁 - 写
控制主体对客体的写（或者修改）权限
权限锁 - 删除
控制主体对客体的删除权限
范围锁 - 创建
控制主体对客体的范围创建权限。例如，拥有一个模型库-仓库A的范围创建权限，就可以在该仓库下创建子仓库以及功能单元模型。
范围锁 - 读
控制主体对客体的范围读权限。例如，拥有一个模型库-仓库A的范围读权限，就可以在查询该仓库下的子仓库列表，需要说明的是，如果想查询该仓库内的数据，除了需要有范围读权限，还需要有该仓库内数据的读权限。
范围锁 - 写
控制主体对客体的范围写权限。例如，拥有一个模型库-仓库A的范围写权限，就可以修改该仓库下的子仓库，需要说明的是，如果想修改该仓库内的数据，除了需要有范围写权限，还需要有该仓库内数据的修改（写）权限。
范围锁 - 删除
控制主体对客体的范围删除权限。例如，拥有一个模型库-仓库A的范围删除权限，就可以删除该仓库下的子仓库，需要说明的是，如果想删除该仓库内的数据，除了需要有范围删除权限，还需要有该仓库内数据的删除权限。

# 3.1等保4.0标准

等保四级部分标准要求：
应由授权主体配置访问控制策略，访问控制策略规定主体对客体的访问规则；
访问控制的粒度应达到主体为用户级或进程级，客体为文件、数据库表级；
应对主体、客体设置安全标记，并依据安全标记和强制访问规则确定主体对客体的访问。

# 3.2访问控制

访问控制（Access control）是指对访问者向受保护资源进行访问操作的控制管理。该控制管理保证被授权者可访问受保护资源，未被授权者不能访问受保护资源。
现实生活中的访问控制可以由付费或者认证达成。例如：进电影院看电影，需要购买电影票，否则检票员就不让你进去。
访问控制有很多模型，比如：
l 自主访问控制模型（DAC：Discretionary Access Control）
l 强制访问控制模型（MAC：Mandatory Access Control）
l 角色访问控制模型（RBAC：Role-based Access Control）
l 属性访问控制模型（ABAC：Attribute-Based Access Control）
在WOS4.0中，用到了强制访问控制模型和角色访问控制模型，前者通过级别和类别（安全标记）校验，后者通过用户身上的钥匙校验。
自主访问控制：细粒度、灵活性高、配置效率低；强制访问控制：控制粒度大、灵活性不高、安全性强。

# 3.3RBAC3

RBAC是**基于角色的访问控制**（Role-Based Access Control）在RBAC中，权限与角色相关联，用户通过成为适当角色的成员而得到这些角色的权限。这就极大地简化了权限的管理。这样管理都是层级相互依赖的，权限赋予给角色，而把角色又赋予用户。
RBAC级别：RBAC0 ~ RBAC3。
RBAC0：是RBAC的核心思想；

RBAC1：是把RBAC的角色分层模型；引入了角色继承概念，有了继承就有了上下级的包含关系；

- 3.先决条件约束：用户想要得到高级权利，必须先拥有低级权力。

​ DSD：动态决定怎么样计划角色，如一个用户拥有5个角色，只能激活2个。

RBAC3：RBAC1 + RBAC2。

### 3.3RBAC3
