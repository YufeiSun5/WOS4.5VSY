# 2.1.1. 传输网关的安全规则

互联网关存在一个默认仓库，是其启动时自动生成，用户可以为该仓库挂范围锁。
登录运维管理工具的用户需要有对应的仓库范围锁权限后，才能够配置数据映射信息，方可完成映射的管理。
传输接收侧，接收数据的用户应具备接收数据相应的权限，此时才能完整的完成传输任务。

### 2.1.1. 传输网关的安全规则

# 2.1.2. 安装运维的审计规则

传输网关对于映射关系的管理写操作，均需要记录审计，审计策略无需用户配置，默认全审计。该审计记录需要传输网管处理，详情参见审计产品定义
传输过程产生的审计，为各个子系统触发的审计，用户为接收云配置的接受用户。

# 3.5.2. 集群部署

​ 传输网关集群保证传输系统的可靠性。传输任务可分散到不同的实例上执行。某个节点挂掉，任务转移到其他节点执行。
​ 接收节点如果挂掉，发送方连接其他节点重新传数据。
​ 多副本模式下，传输网关需要向用户层面暴露虚拟IP，虚拟IP由系统运维配置。

### 3.5.2. 集群部署

# 3.6. 互联网关事务

附：
 事务的四大特性
 1.原子性
 确保事务是一个不可分割的操作单位，事务中的所有操作要么全成功执行，要么全部失败并回滚，没有处于中间状态的可能性；
 2.一致性
 事务执行的结果必须是使数据库从一个一致性状态变到另一个一致性状态，事务执行过程中对数据库的修改必须满足一定的约束条件，以保证数据的完整性和有效性。
 3.隔离性
 确保并发执行的事务之间相互隔离，互不干扰，每个事务都应该感觉不到其他事务的存在，即使多个事务并发执行，也不能产生相互干扰的效果。
 4.持久性
 一旦事务提交成功，对数据库的修改就是永久性的，即使在系统故障的情况下，也能够保证数据的持久性，不会丢失已提交的实务操作结果。
事务的隔离级别
 5.读未提交
 一个事务可以读取另一个未提交事务的修改，可能出现脏读、不可重复读和幻读问题。
 6.读已提交
 一个事务只能读取另一个已提交事务的修改，可防止脏读，但可能出现不可重复读和幻读问题。
 7.重复读
 在一个事务中多次读取同一个数据时，能够保证读取到的数据一致，即使其他事务修改了该数据，可防止脏读和不可重复读，但可能出现幻读问题。
 8.序列化
 它通过强制事务串行执行，从而避免了脏读、不可重复读和幻读问题，但会降低并发性能。
事务中常见的问题：
 9.脏读：
 当我们处于读未提交事务隔离级别是，一个事务可以读到另一个事务未提交的数据，但若这条未提交的数据进行了回滚，就会导致刚刚读到的数据有问题，也就是读到了脏数据，而这时，我们若是使用这条脏数据进行更新操作，就会导致脏写。
 10.不可重复读：
 当我们处于读已提交的隔离级别，在同一个事务内，我们在不同时间内查询得到的数据结果是不同的，这就是不可重复读，导致的原因，可能是另一个食物在这个实务操作的过程中完成了对于数据的更新操作。
 11.幻读：
 在可重复的隔离界别下，事务在读取数据后，其他事务不能修改或删除这些数据，但可以插入系的数据，导致幻读问题的发生。
