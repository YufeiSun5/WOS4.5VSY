# 方案设计

## 目标和用户场景

- 目标：设计一个新的 WOS4 测试页，代号 `盛云科技_孙宇飞_Palimpsest`，用于验证左侧菜单页面、右侧内容页面、假数据 CRUD、图表和弹出页。
- 使用者：孙宇飞及后续参与实施的 frontend-ai、code-ai、test-ai。
- 业务场景：类似“实习生考核管理 / 设备台账管理 / 测试数据管理”的后台管理页。
- 成功标准：
  - 首屏结构清晰，左侧菜单固定，右侧业务区可切换。
  - 查询、新增、编辑、删除、查看/评分的交互路径完整。
  - 数据结构和后端函数契约足够明确，后续 code-ai 可直接实现。
  - 所有未验证平台能力都标记为待实测，不误写成已完成。

## 输入材料

- 草图 / 截图 / 设计稿：`assets/参考图-人才管理考核页.png`
- 前端交互说明：左侧菜单页面 + 右侧内容页面；不设置顶部菜单或顶部栏。右侧内容页内部包含筛选、统计、图表、表格、弹出页。
- 后端数据结构：本方案定义 `AssessmentRecord` 假数据和 CRUD 函数。
- 已有数据或已有页面：clean CRUD 历史页 `查询列表`、Nadir 蓝色客户端对象、`DemoCrudBackend_CUSTOMFUNC` 相关记录。
- 用户补充说明：本轮“先设计”，不要求立即实施。
- 用户追加判断：大概率可以让一个客户端左侧显示菜单页面，右侧显示内容页面。
- 用户追加约束：不用顶部菜单，只有左右两部分。

## 现有系统核实

### 已确认存在的页面 / 客户端

- 组态系统客户端可创建客户端和画面，并通过 `数字孪生可视化` 添加/引用页面精灵图。
- 蓝色 WebJS 客户端对象已有可运行路径，正式视图可以挂载页面 iframe。
- 正式客户端运行时要优先检查 inner `.page_view.__vue__`，不能只看顶层壳。
- 用户确认双页面并排是高置信方向，但仍需要 AI 在正式客户端运行态补证。

### 已确认存在的组件 / 布局

- 已验证：`RContainer/RRow/RCol` 布局、输入框、按钮、表格、折线图。
- 只探索过或未完成当前主线验收：树、文本、导航菜单、选择器、日期选择器、平台弹窗组件。
- 因此第一版设计应优先使用已验证组件；左侧菜单可用一列按钮/文本项模拟，不先依赖未验证的导航菜单组件。

### 已确认存在的模型 / 后端函数

- 历史 clean CRUD 后端已能创建 `QueryRecords / InsertRecord / UpdateRecord` 等自定义函数并保存提交。
- `PageView.eval` 环境中曾验证 `Call(QueryRecords)` 可返回 `ret=0`。
- 可见按钮 Click 脚本仍曾返回 `-210133`，说明正式按钮执行上下文还需要 code-ai 专项修复和验证。

### 已确认存在的数据结构 / 接口

- 页面变量模式已验证：`seedData`、`filterKey`、`refreshCounter`。
- 后端 `QueryRecords` 使用 `stringMap<var>` 入参和 `stringMap<var>` 返回值的方向已明确。
- 真正的数据库持久化 CRUD 尚未作为稳定能力验收，本次先设计函数契约和假数据结构。

### 已确认存在的测试或历史任务证据

- `wos4-button-variable-flow` 记录了按钮变量联动和后端 Call 当前边界。
- `wos4-config-client-screen-create` 记录了客户端画面添加、首页、正式预览和 inner runtime 检查。
- `wos4-blue-client-object-create` 记录了蓝色 WebJS 客户端对象创建和视图验证。

## 对“左右两个页面”的判断

当前可给出的准确结论：

- 可以实现“客户端上看起来是左侧菜单 + 右侧页面”的体验，推荐用同一个 WOS4 页面精灵图内部的左右分栏实现。
- 可以在客户端里挂载多个画面并设置首页，但这只能证明客户端支持多画面管理和切换，不等于证明同一视口内能同时并排显示两个独立页面对象。
- 用户明确判断“一个客户端左侧显示菜单页面、右侧显示内容页面”大概率可行，因此后续实施应优先验证这一路径。
- 本项目当前还缺少 AI 保存的正式运行证据来证明“左侧一个独立页面对象、右侧另一个独立页面对象，同时显示在一个客户端视口里”已经稳定可交付。

推荐把“双独立页面并排”作为第一验证目标，但设置明确降级路径。目标结构：

```text
客户端 Palimpsest
├─ 左侧区域：PalimpsestMenu 页面
└─ 右侧区域：PalimpsestContent 页面
```

该结构没有独立顶部菜单；右侧页面如需标题、筛选或操作按钮，全部放在右侧内容页面内部。

如果验证发现平台不能稳定持久化双页面并排，则降级为单页面 shell：

```text
一个页面精灵图
├─ 左侧菜单区
└─ 右侧内容区
   ├─ 顶部筛选
   ├─ KPI / 图表
   ├─ 表格
   └─ 弹出层 / 抽屉
```

双页面并排实施前必须专项验证：

```text
客户端画面 A：Shell/菜单
客户端画面 B：内容页
验证是否能把 B 嵌入 A 的右侧容器，或只能通过 SetPage/首页机制切换
```

## 缺口和风险

- 待核实内容：
  - `PalimpsestMenu` 和 `PalimpsestContent` 是否能在同一个客户端视口内左右并排显示。
  - 导航菜单组件是否能稳定持久化。
  - 平台弹窗组件是否可在页面精灵图中稳定保存和正式运行。
  - 表格行内操作按钮和行选择事件是否可稳定触发。
  - 可见按钮脚本中的 `Call(...)` 是否能在蓝色客户端对象中稳定返回 `ret=0`。
- 缺失内容：
  - 本任务尚未创建新页面、新后端模型和新客户端对象。
  - 尚未创建真实可持久化数据库表。
- 技术风险：
  - 双页面并排如果依赖平台内部页面容器，可能比单页面分栏复杂且不可控。
  - 弹窗如果依赖未验证组件，可能提交后丢失或正式运行不显示。
  - 后端 CRUD 如果绕过正式客户端运行上下文验证，容易出现编辑器可行、正式按钮失败。
- 需要用户确认的问题：
  - 是否接受第一版用单页面分栏模拟左菜单 + 右页面。
  - 弹窗是否可先做同页遮罩/抽屉。
  - CRUD 第一阶段是否允许先假数据，第二阶段再真实后端持久化。

## 推荐方案

- 推荐路径：
  1. 新建或规划客户端 `盛云科技_孙宇飞_Palimpsest`。
  2. 优先验证双页面结构：左侧 `PalimpsestMenu`，右侧 `PalimpsestContent`。
  3. 如果双页面并排验证失败，改为新建一个页面精灵图，命名建议 `盛云科技_孙宇飞_Palimpsest -> 综合管理测试页`，采用单页左右分栏。
  4. 使用已验证组件：输入框、按钮、表格、折线图；KPI 卡片可先用文本/容器样式实现，若文本组件不稳则用表格/按钮文案降级。
  5. 后端设计 `AssessmentRecord` 数据结构和 `QueryRecords/CreateRecord/UpdateRecord/DeleteRecord` 函数。
  6. 弹出页优先做同页遮罩；实施中若平台弹窗不可持久化，降级为右侧抽屉。
  7. 正式验收必须走蓝色 WebJS 客户端对象的可见按钮点击，不只接受编辑器预览。
- 不推荐路径：
  - 第一版直接追求两个独立页面对象并排显示。
  - 第一版依赖未验证导航菜单、日期选择器、复杂图表和行内按钮。
  - 只在 raw iframe window 中调用后端并据此判断成功或失败。
- 依赖条件：
  - 需要 frontend-ai 按布局 skill 先完成布局再放组件。
  - 需要 code-ai 按后端 skill 创建或复用自定义函数，并记录提交版本。
  - 需要 test-ai 在正式客户端中做点击前后对比。
- 备选方案：
  - 如果弹窗失败：改为右侧详情抽屉。
  - 如果导航菜单失败：用按钮列表模拟菜单。
  - 如果后端 CRUD 暂时无法持久化：先使用页面变量 mock，保留后端接口契约。
